## 2. 기술 스택

프로젝트에서 확정된 스택을 기준으로 하며, 프론트엔드 개발 방식과 LLM 모델은 이번 버전에서 새로 확정했다.

```
[프론트엔드: Next.js + Tailwind]  ──▶ Vercel 배포
   ▲  (Google Stitch로 생성한 UI 문서 기반으로 구현)
   │ API 호출
   ▼
[백엔드: FastAPI + ChromaDB]  ──▶ Railway 또는 Render 배포 (Vercel 서버리스와 분리)
   │
   ├──▶ LLM API: OpenAI GPT-5.4-mini (해석/선별/설명 생성)
   └──▶ Supabase(Postgres): user_profile / hobby_profile / feedback_history / business_event 저장
```

| 레이어 | 기술 | 비고 |
|---|---|---|
| 프론트엔드 | Next.js + Tailwind CSS | Google Stitch로 생성한 UI 문서(화면 설계)를 기준으로 구현. 화면 컴포넌트/레이아웃은 Stitch 산출물을 소스 오브 트루스로 사용 |
| 프론트 배포 | Vercel | |
| 백엔드 | FastAPI | LLM 호출·추천 파이프라인·API 오케스트레이션 |
| 벡터 DB | ChromaDB | Railway/Render 상시 서버로 배포(Vercel 서버리스와 분리해 데이터 유실 방지) |
| 관계형 DB | Supabase (Postgres) | 정형 데이터 저장 + 어드민 최소 기능(hobby_profile 조회/수정, feedback_history 로그) 겸함 |
| LLM | OpenAI GPT-5.4-mini (gpt-5.4-mini) | 자유입력 심층분석(②), 후보 선별·설명 생성(④). 사양: 400K 컨텍스트, 입력 $0.75 / 출력 $4.50 per 1M 토큰 (OpenAI, 2026-03-17 출시). 텍스트+이미지 입력, function calling, structured output 지원 |
| RAG 리트리버 | LangChain (최소 사용) | RAG 검색부에만 사용. LLM 해석/판단/설명 생성은 OpenAI SDK 직접 호출 |
| 임베딩 | OpenAI 임베딩 API | hobby description + 핵심 태그를 임베딩 대상으로 사용 (임베딩 모델명은 착수 시점에 최신 버전 확인 필요) |

개발 제약 원칙: 개발 기간 2일, 비전공자 팔로우업 고려. 코드 대신 프롬프트·데이터 시트 수정만으로 로직을 바꿀 수 있게 설계한다. 랭킹 스코어 산식·전역 학습·협업 필터링·컨텍스추얼 밴딧·자체 ML 학습은 MVP에서 배제한다.

## 3. 문제 정의

### 3-1. 핵심 문제

대한민국 1인 가구 20대의 주말 여가는 영상시청(75.7%)에 압도적으로 쏠려 있고, 취미·자기개발 활동은 17.8%에 그친다. 리서치 결과 이 격차의 원인은 흥미 부재가 아니라 "나한테 맞는 걸 걸러줄 사람이 없다"는 결정 피로였다.

핵심 리서치 발견:

| 발견 | 내용 | 제품 함의 |
|---|---|---|
| "취미없음" 방어기제 | 스스로를 "재미없는 사람"으로 자평 | 온보딩은 진단형이 아니라 안심형("취미는 천천히 발견하면 됩니다") |
| 모임형 신뢰 결핍 | 문토·소모임 후기에 보증금 갈취, 위장모임, 성비 불균형 우려 다수 | 오프라인 사회성 추천에는 안전근거 문구 필수 |
| 선택지 과잉 | 큐레이션 신뢰는 1회성이 아니라 "지속적으로 알아가는 관계"에서 형성 | Step1부터 "점점 알아간다"는 체감 설계 |
| 번아웃 ≠ 보어아웃 | 과로형 소진과 권태형 무기력은 회복 접근이 다름 | 자유입력에서 "지쳤다" vs "지루하다/허무하다" 구분 |
| 비용 막연한 불안 | "무료로 먼저 시작 가능한지"가 실행률을 좌우 | 비용은 "무료로 먼저 확인 가능" 프레임 우선 |

### 3-2. 핵심 가치 제안

리스트가 아니라 "처방"이다. 취미로는 구조화 설문과 자유입력 심층분석으로 개인의 성향·제약·오늘의 상태를 파악하고, 취미 3~5개를 "왜 나에게 맞는지 + 오늘 당장 할 수 있는 5~10분 첫 행동 + 어디서 시작하는지" 근거와 함께 제시한다.

클래스101·프립·문토·소모임 같은 실행 플랫폼과 경쟁하지 않고, 사용자를 그들에게 연결하는 발굴·처방 레이어로 포지셔닝한다.

- 1순위 차별화: 대화형 개인화 처방 — "나를 이해한 추천"이라는 체감. 모임 신뢰 결핍이라는 시장 공백을 안전근거 문구로 직접 공략한다.
- 핵심 차별점: "의외의 발견". 성향에 안정적으로 들어맞는 추천만 주는 서비스들과 달리, 취미로는 사용자가 스스로 떠올리지 못했을 의외 추천을 모든 사용자에게 항상 함께 제시한다. 일반(안정) 추천과 의외 추천을 화면에서 명확히 구분해, "나를 이해한 안전한 추천"과 "나를 확장하는 뜻밖의 추천"을 동시에 경험하게 한다. 이 "발견" 정체성이 단순 매칭 서비스와의 결정적 차이다.

### 3-3. Non-Goals (이번 MVP에서 하지 않는 것)

- 랭킹 스코어 산식 기반 추천, 전역 학습, 협업 필터링, 컨텍스추얼 밴딧, 자체 ML 학습
- 행동 기반 5~10분 미니체험 (→ Step5 체험 후 체크인으로 대체)
- 결제·예약 직접 연동, 실제 리퍼럴 정산, 외부 플랫폼 API 연동
- 자동 푸시 알림 기반 체크인 (재방문은 완전 user-initiated)
- 수동 신호 연동(음악/영상/사진/캘린더/소비 카테고리)
- 커뮤니티 기능, B2B 인사이트 판매, 파트너 대시보드
- 취미 자체를 판매하거나 클래스를 직접 운영하는 것
- 번아웃·감정 상태에 대한 의학적 진단 (참고용 컨디션 값으로만 활용)

## 4. 사용자 정의 (User Persona)

본 MVP는 페르소나를 1명으로 제한해 설계·검증한다.

### 4-1. 페르소나: 지친 마케터 "이서연"

| 항목 | 내용 |
|---|---|
| 나이·성별 | 26세, 여성 |
| 직업 | 스타트업 마케터, 야근이 잦음 |
| 주거 | 원룸 1인가구 |
| 상태 | 시간이 부족하고 지쳐 있음. 주말엔 거의 방전 상태 |
| 관계 | 관계 결핍은 있지만, 모임 실패 경험(어색함·부담)에 대한 불안이 큼 |
| 심리 | "번아웃 진단"을 받는 느낌은 원하지 않음. 취미 찾기를 여러 번 시도했지만 실패 |
| 예산·시간 | 월 여가비 1만원 이하 감각, 세션당 30분 이내를 선호 |

페인포인트:
- 정보(앱·블로그·클래스)는 넘치는데 "나한테 맞는 것"을 걸러줄 사람이 없다.
- 선택지가 너무 많아 결정 자체를 못 내리고, 결국 침대에서 폰만 본다.
- 모임형 취미는 관심 있지만 "낯선 사람과 큰 모임"은 부담스럽고 불안하다.
- "무료로/부담 없이 먼저 해볼 수 있는지"가 실행 여부를 가른다.

대표 상황(JTBD): "평일 밤 방전 상태로 침대에 누워, 뭔가 하고는 싶은데 뭘 할지 모르겠을 때 — 나를 이해하고 부담 없이 시작할 거리를 딱 골라줬으면 좋겠다."

## 6. 유저 스토리 및 인수 기준

### US-1. 전체 경로 (설문 → 추천 → 피드백 재구성 → 체크인)

지친 마케터로서, 나는 부담 없이 설문에 답하는 것만으로 내 상태에 맞는 취미를 처방받고, 마음에 들지 않으면 다시 맞춰볼 수 있으며, 나중에 잘 지내고 있는지 확인받고 싶다. (참조: 테스트 사용자 TU01)

인수 기준 (AC):
- [ ] 설문 완료 후 추가 대화 턴 없이 바로 Step3 추천이 생성되며, 총 소요는 2분 내외다.
- [ ] 자유입력이 비어있거나 무의미하면 설문 데이터만으로 추천을 생성한다.
- [ ] 추천 카드에 정량 근거와 정성 근거(관계 결핍 맥락 등)가 함께 표시된다.
- [ ] 번아웃 신호가 있어도 의외 추천은 억제되지 않고 별도 "뜻밖의 발견" 영역에 항상 노출된다. 단 그 의외 추천도 완화된 하드 실행 제약(제약치의 최대 1.5배, FR-407-1)은 지킨다.
- [ ] "이미 해봤어요" 선택 시 부정으로 처리하지 않고 유사 변형 취미로 재신호한다.
- [ ] "부담스러워요" 피드백 시 원인 축 질문 1개 후 1회 재구성되고, 이후 추가 재구성은 차단된다.
- [ ] 온라인 사회성 추천 카드에는 안전근거 문구가 표시되지 않는다.
- [ ] 재방문 시 설문 반복 없이 "오늘의 상태"만 물은 뒤 체크인이 진행된다.

### US-2. 안전 분기 (risk_flag)

심한 무기력을 겪는 사용자로서, 나는 취미 추천이 아니라 공감과 도움 안내를 먼저 받고 싶다. (참조: TU03)

인수 기준 (AC):
- [ ] 자유입력에서 심한 소진/무기력 신호 감지 시 risk_flag=true가 산출된다.
- [ ] 추천 파이프라인(③~⑤)이 실행되지 않고 공감 응답 + 전문가/상담 안내로 전환된다.
- [ ] 어떤 Step에서 신호가 감지되든 동일하게 안전 분기가 강제된다.

### US-3. 비용 안심

비용에 막연한 불안이 있는 사용자로서, 나는 "무료로 먼저 해볼 수 있는지"를 카드에서 바로 확인하고 싶다.

인수 기준 (AC):
- [ ] 카드 비용 영역에 "예상 범위" 고지가 함께 표시된다.
- [ ] 확정 가격 단정 표현이 노출되지 않는다.
- [ ] 첫행동은 예약·결제·가입 없이 5~10분 내 완결 가능한 행동으로만 제시된다.

## 7. 비기능 요구사항 (Non-Functional Requirements)

### 7-1. 성능 요구사항 — 추천 생성 응답 시간

원본 목차의 "계약서 분석 응답 시간"을 본 서비스의 핵심 처리인 추천 생성 응답 시간으로 치환. 아래 목표치는 초기 목표이며 부하 테스트로 보정 필요.

| 지표 | 목표 | 비고 |
|---|---|---|
| Step1 → Step3 추천 생성 | p95 ≤ 6초 | ②LLM 심층분석 + ③검색 + ④선별·설명 + ⑤RAG 합산 |
| Step4 재구성(1회) | p95 ≤ 5초 | ②~④ 재실행 |
| 스트리밍 첫 토큰(TTFT) | ≤ 1.5초 | 설명 생성 시 스트리밍으로 체감 지연 완화 권장 |
| 벡터 검색 단독 | ≤ 300ms | ChromaDB 상시 서버 |

### 7-2. 보안 요구사항

| 항목 | 요구사항 |
|---|---|
| 데이터 전송 암호화 | 모든 클라이언트-서버, 서버-외부 API 통신은 TLS(HTTPS)로 암호화한다 |
| 사용자·감정 데이터 보관 (원본 "계약서 파일 보관" 치환) | 감정·번아웃 데이터(energy_level, burnout_state, current_emotion)는 민감정보로 분류해 식별 정보와 별도 테이블로 분리 저장하고 결합 조회를 최소화한다. 최소 수집·명시적 동의 원칙을 적용한다. 자유입력 원문은 필요 목적 외 장기 보관하지 않는다(보관 기간 정책 별도 수립 필요) |
| 개인정보 | 개인정보 최소 수집. 진단형 필드명(번아웃 점수, 우울 진단) 사용 금지 → 참고형 명명(energy_level, burnout_state) 유지. 국내 개인정보보호법 준거(수집·이용 동의, 민감정보 별도 동의) — 법무 검토 필요 |
| API Key 관리 | OpenAI·Supabase 등 키는 환경변수/시크릿 매니저로 관리하고 프론트엔드에 노출하지 않는다. LLM 호출은 반드시 백엔드(FastAPI) 경유. Vercel/Railway 시크릿 저장소 사용, 저장소 커밋 금지 |

### 7-3. 가용성 및 확장성 요구사항

| 항목 | 요구사항 |
|---|---|
| 장애 복구 시간 | ChromaDB/Supabase는 매니지드 백업을 사용하고, ChromaDB는 Vercel 서버리스와 분리 배포해 재기동 시 벡터 데이터 유실을 방지한다. 목표 복구 시간(RTO) ≤ 1시간(MVP 기준, SLA 아님) |
| LLM 장애 대응 | 외부 LLM/임베딩 API 장애 시 사전 저장된 정적 추천 세트(RIASEC 유형별 대표 취미 6~12개)로 대체 응답한다(상세는 8-2) |
| 확장성 | MVP는 단일 페르소나·소규모 트래픽 전제. 파이프라인은 stateless하게 설계해 수평 확장 여지를 남긴다. 대규모 개인화·전역 학습은 v2 |

## 8. AI 특화 요구사항 (AI-Specific Requirements)

### 8-1. 할루시네이션 방지 전략

본 서비스에서 할루시네이션이 사용자 신뢰에 직접 타격을 주는 지점은 취미 비용·안전 정보다(리서치의 "비용 막연한 불안" 발견과 직결). 발생 자체를 억제하되, 발생 시에도 사용자에게 잘못된 사실이 전달되지 않도록 이중으로 막는다.

**발생 억제 (사전)**

| 대상 | 전략 |
|---|---|
| 비용(cost_range)·안전(safety_notes) | AI 생성 금지. 사람이 검증한 값만 지식베이스에 저장하고, 카드에는 RAG로 조회한 값만 노출한다. LLM이 임의로 가격·주의사항을 생성하지 못하도록 프롬프트에서 명시 차단 |
| 취미 실행 정보(장비/난이도/시간) | RAG 지식베이스 조회값을 근거로만 제시. 지식베이스에 없으면 "확인 필요"로 표기하고 단정하지 않는다 |
| 근거문장·첫행동 | 후보로 넘어온 취미 데이터 범위 내에서만 생성(사전 구축한 고정 hobby_profile 밖의 취미는 추천 불가) |
| 감정·번아웃 문구 | 진단형 표현 금지 규칙을 프롬프트에 포함하고 참고형 표현만 허용 |

**발생 시 대응 (사후)**

| 상황 | 대응 |
|---|---|
| LLM 응답이 JSON 스키마를 벗어남 | 1회 재요청 → 재실패 시 설문 태그 매칭만으로 추천 생성(임베딩·LLM 설명 생략) |
| 카드에 확정 가격처럼 단정한 표현 감지 | 후처리 검증에서 "예상 범위" 고지 강제 삽입 |
| 추천 취미가 지식베이스에 없음 | 해당 카드 노출 차단(존재하지 않는 취미 생성 방지) |
| 사용자 신고/오류 리포트 | 어드민에서 해당 hobby_profile 즉시 수정(시트 수정 → 재적재) |

### 8-2. Fallback 전략

| 상황 | 대응 |
|---|---|
| LLM 응답 JSON 스키마 미준수 | 재요청 1회 → 실패 시 설문 태그 매칭만으로 추천 생성(임베딩·LLM 설명 생략) |
| 자유입력이 비어있거나 무의미 | 설문 데이터만으로 추천 생성 진행 |
| 벡터 검색 결과 0건 | RIASEC 코드 매칭 기준으로 후보 완화 검색 |
| 외부 API(임베딩/LLM) 장애 | 사전 저장된 정적 추천 세트(RIASEC 유형별 대표 취미 6~12개)로 대체 응답 |
| 심한 소진·자해 신호 감지 | 추천 로직 즉시 중단, 공감 응답 + 전문가/상담 안내로 대체(최우선, 다른 fallback보다 상위) |

### 8-3. 비용·속도 예산

**모델 단가 (GPT-5.4-mini, OpenAI 2026-03-17 출시 기준)**

| 항목 | 값 |
|---|---|
| 입력 | $0.75 / 1M 토큰 |
| 출력 | $4.50 / 1M 토큰 |
| 컨텍스트 | 400K 토큰 |

**세션당 LLM 호출 예산 (개략 추정 — 실측으로 보정 필요)**

MVP 파이프라인에서 GPT-5.4-mini 호출은 세션당 ② LLM 심층분석 1회 + ④ 선별·설명 1회(+ Step4 재구성 발생 시 추가). 임베딩·벡터검색·RAG 조회는 LLM 생성 대비 비용이 미미하다.

| 시나리오 | LLM 호출 | 대략 토큰(입력+출력) | 개략 비용 |
|---|---|---|---|
| 기본 경로 | ② + ④ = 2회 | ~2K in / ~1.5K out | 약 $0.008 / 세션 |
| 재구성 포함 경로 | ② + ④ + 재구성(②~④) = ~4회 | ~4K in / ~3K out | 약 $0.015 / 세션 |

위 수치는 토큰 가정에 기반한 개략치다. 실제 프롬프트·컨텍스트 길이에 따라 달라지므로 파이프라인 통합 테스트 시 실측으로 갱신한다.

**속도·비용 절감 원칙**

- 설명 생성은 스트리밍으로 체감 지연을 낮춘다.
- 프롬프트에 불필요한 컨텍스트를 넣지 않는다(후보 취미는 3~5개 데이터로 제한).
- Step4 재구성은 1회로 제한해 폭주를 막는다(기능 제약이자 비용 제약).
- risk_flag 분기 시 후속 LLM 호출을 즉시 중단해 불필요한 비용/지연을 없앤다.
- 반복되는 정적 응답(fallback 세트)은 캐시/사전 저장으로 LLM 호출 없이 서빙한다.

     └─ Step5 체크인 → user_profile 장기 필드 갱신
  ※ ② risk_flag=true 시 ③~⑤ 미실행 → 공감 + 상담 안내로 분기
