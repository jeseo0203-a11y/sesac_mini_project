# AGENTS.md — 할래말래(HalaeMalae) 취미 추천 AI 에이전트

이 문서는 이 저장소에서 작업하는 코딩 에이전트(Claude Code 등)를 위한 규칙서다.
`할래말래_PRD_v1.2.md`, `SKILL.md`, `HobbyDNA_9축_클릭선택_설문설계.md`, `hobby_profile_dataset_v1.json`을 단일 진실 원천으로 삼는다.
**규칙과 다른 판단이 필요한 순간에는 임의로 결정하지 말고 먼저 사용자에게 확인한다.**

---

## 1. 서비스 한 줄 요약

"할래말래"는 구조화 설문(9축) + 적응형 대화로 사용자의 성향·제약·오늘의 상태를 파악해,
취미 3~5개를 **"왜 나에게 맞는지 + 오늘 당장 할 수 있는 5~10분 첫 행동 + 어디서 시작하는지"**
근거와 함께 처방하는 서비스다. 리스트가 아니라 **처방**이며, 랭킹 스코어·전역 학습·ML은 쓰지 않는다.

- MVP 페르소나는 **1명으로 고정**: 26세 여성, 야근 잦은 스타트업 마케터, 원룸 1인가구, 번아웃 상태, 월 여가비 1만원 이하 감각, 세션당 30분 이내 선호.
- 핵심 차별점(협상 불가): **"의외의 발견"** — 모든 사용자에게 의외 추천 1개를 항상, 게이팅 없이 노출. 단 예산·시간·공간 같은 하드 실행 제약은 의외 추천에도 동일 적용.

## 2. 기술 스택 (확정 — 임의 변경 금지)

```
프론트: Next.js + Tailwind CSS → Vercel 배포 (화면은 Google Stitch UI 문서가 소스 오브 트루스)
백엔드: FastAPI → Railway/Render 배포 (Vercel 서버리스와 분리, ChromaDB 데이터 유실 방지)
벡터DB: ChromaDB
관계형DB: Supabase (Postgres) — user_profile / hobby_profile / feedback_history / business_event
LLM: OpenAI GPT-5.4-mini (gpt-5.4-mini) — OpenAI SDK 직접 호출
RAG 리트리버: LangChain "최소 사용" (검색부에만) — LLM 해석/판단/설명 생성은 LangChain 미사용, SDK 직접 호출
임베딩: OpenAI 임베딩 API (모델명은 착수 시점 최신 버전 재확인 필요)
```

**금지 사항**
- LangChain을 LLM 호출/RAG 전반에 무겁게 끌어오지 말 것 (검색부 최소 사용 외 금지).
- `pymupdf`, `pdfplumber`, `python-multipart` 사용 금지 (계약서 분석 템플릿 잔재, 파일 업로드 기능 없음).
- 랭킹 스코어 산식, 전역 학습, 협업 필터링, 컨텍스추얼 밴딧, 자체 ML 학습 코드 작성 금지.
- API 키(OpenAI/Supabase)는 환경변수/시크릿 매니저로만 관리. 프론트엔드 노출 금지. LLM 호출은 반드시 백엔드 경유.

**설계 원칙**: 로직 변경은 최대한 **코드가 아니라 프롬프트 또는 데이터 시트(`hobby_profile`) 수정**으로 처리 가능하게 만든다 (개발 기간 2일, 비전공자 팔로우업 고려).

## 3. 절대 금지 / 최우선 규칙

### 3-1. 톤 & 표현
- 서비스 페르소나는 "취미 처방사". 따뜻하고 공감하는 말투, 나이·성별 무관하게 편안하게.
- **절대 금지 단어**: "진단", "점수", "장애", "우울증" 등 의학적 표현. 전 Step UI에서 일관 금지.
  - 대체 표현: "오늘의 컨디션", "에너지 상태", "회복 필요도".
- `beginner_action`(첫걸음 행동)은 **5~10분 내 실행 가능한 것만**. 예약·결제·가입 등 고관여 행동 제안 절대 금지.

### 3-2. risk_flag 안전 분기 — 예외 없이 최우선
자유입력·대화·Step5 자유응답 어디서든 자해/극단적 무기력 신호가 감지되면:
1. `emotional_state = "risk"`, `risk_flag = true` 산출.
2. 이후 파이프라인(③ 후보검색 → ④ 선별·설명 → ⑤ RAG 보강)을 **즉시 중단**한다. 어떤 상황·어떤 fallback보다 상위 규칙.
3. `support_message`는 아래 고정 문구를 **변형 없이 그대로** 반환한다:
   > "당신의 마음을 들어줄 전문가들이 늘 기다리고 있습니다. 정신건강상담전화 [109] 또는 보건복지상담센터 [129]로 연락해 보세요."
4. 케이스 B(JSON)에서는 `recommendations: []`, `beginner_action: ""` 이어야 한다.

### 3-3. 할루시네이션 방지
- `초기비용`/`지속비용`/`safety_notes(안전)`는 **LLM이 절대 생성하지 않는다**. 데이터셋에 사람이 검증한 값만 사용 (프롬프트에 명시적 차단 문구 포함).
- 카드 비용 표기는 항상 "예상 범위이며 실제와 다를 수 있어요" 고지를 동반한다.
- `hobby_id`/`hobby_name`은 `hobby_profile` 데이터셋에 존재하는 값만 사용. 존재하지 않는 취미를 만들어내지 않는다.
- 추천 취미가 지식베이스에 없으면 해당 카드 노출을 차단한다(생성 방지).
- LLM 응답이 JSON 스키마를 벗어나면 1회 재요청 → 재실패 시 설문 태그 매칭만으로 추천 생성(임베딩·LLM 설명 생략).

```json
{

    "emotional_state": "burnout_type | boreout_type | steady",

  },
  "recommendations": [
    {
      "hobby_id": "string(데이터셋 그대로)",
      "hobby_name": "string(데이터셋 그대로)",
      "reason_clause": "string",
      "recommendation_type": "안정 | 의외 | 회복 | 관계"
    }
  ],
  "beginner_action": "string",
  "risk_flag": false,
  "support_message": ""
}

// 케이스 B: risk 상태
{
  "user_status": { "emotional_state": "risk", "empathy_message": "string" },
  "recommendations": [],
  "beginner_action": "",
  "risk_flag": true,
  "support_message": "당신의 마음을 들어줄 전문가들이 늘 기다리고 있습니다. 정신건강상담전화 [109] 또는 보건복지상담센터 [129]로 연락해 보세요."

```

출력 형식 규칙:
- 반드시 유효한 **단일 JSON 객체**만 반환.
- 마크다운 코드블록 금지, 후행 쉼표 금지.
- `recommendations`는 **반드시 `recommendation_type: "의외"` 카드를 1개 포함**해야 한다(FR-404/407, 협상 불가).

## 5. 데이터셋 (hobby_profile_dataset_v1.json)

- **한글 필드명을 그대로 사용**한다. 영문으로 임의 변환 금지.
```
hobby_id, 취미명, riasec_code, 초기비용, 지속비용, 세션당시간, 공간, 사회성, 강도,
학습곡선, 성취방식, 감각태그, 신규성, beginner_action(첫행동), 준비물,
safety_notes(안전), 외부시작경로, RAG설명(임베딩본문), RAG검색키워드,
is_synthetic, 비용검증상태, 안전검증상태, fallback_set
```
- 값 도메인(실측 기준, 임의 확장 금지):
  - 초기비용: `무료 / 1만원 이하 / 3만원 이하 / 3만원 이상`
  - 세션당시간: `5분 / 15분 / 30분 / 1시간 이상`
  - 사회성: `완전 개인 / 온라인 느슨한 연결 / 오프라인 소규모` (※ "오프라인 대규모"는 현재 데이터 0건 — 필터 로직에서 참조 시 후보 0건 가능성 주의)
  - 강도: `정적 / 저강도 / 중강도 이상` (※ "중강도 이상"은 현재 1건뿐 — 활동적 사용자는 후보 부족 위험, §7 완화 체인 필수 적용)
- 취미 40개 = RIASEC 6유형 균형(R7/I7/A8/S6/E6/C6) + 26세 야근 마케터 페르소나 편향(A유형 강화, 저비용·저시간·실내·개인 취미 다수, 오프라인 소규모 사교는 소수+안전근거 필수).
- **`is_synthetic=TRUE`인 행은 전부 AI 생성 초안**이다. 실데이터와 절대 섞지 말 것 (`FR-1003`).
- `비용검증상태`/`안전검증상태`가 "확인필요"인 동안에는 해당 값을 신뢰 가능한 실데이터처럼 취급하지 않는다. 검증 완료 전까지 프로덕션 노출 여부는 사람 판단이 필요하다.
- `fallback_set=TRUE`인 6개(H001, H008, H015, H023, H029, H035 — RIASEC 유형별 대표)는 **LLM/임베딩 API 장애 시 무조건 보여줄 정적 세트**로, 별도 `fallback_정적세트`에도 고정되어 있다. 하드코딩 캐시로 서빙한다.

## 6. 설문 9축 → 프로필 매핑 (구조화 설문, Step1)

`HobbyDNA_9축_클릭선택_설문설계.md` 기준. 진행 순서: **1단계 핵심 5개 → 2단계 보강 4개** (점진 프로파일링, 텍스트 입력 없이 전부 클릭/버튼).

| 순서 | 단계 | 축 | 비고 |
|---|---|---|---|
| 1 | 핵심 | 에너지·번아웃 (`energy`) | `burnout_severe` 선택 시 초저강도·단기완료·혼자형 자동 상위 노출 |
| 2 | 핵심 | 사회성 (`sociality`) | 5점 스케일 |
| 3 | 핵심 | 활동량 (`activity`) | 4단계 |
| 4 | 핵심 | 시간·낮밤 | 다중선택(A) + 단일선택(B) |
| 5 | 핵심 | 예산 (`budget`) | 4단계, RAG 예상비용과 직접 매칭 |
| 6 | 보강 | 동기(목적) | 최대 2개 선택 |
| 7 | 보강 | 성취 방식 | 4단계, 리포트 설명문에 사용 |
| 8 | 보강 | 공간·오감 | 공간 단일 + 감각 다중(최대 2개) |
| 9 | 보강 | 기질·신규성 | 5점 스케일, 탐색-활용 균형(세렌디피티 비율) 기준값 |

**[필수 추가 — PRD에 없던 SKILL.md 보강 사항]** 위 9축에는 1회 세션 소요시간 문항이 없다. 1단계 핵심 문항에 아래를 반드시 추가할 것:
- 질문: "한 번에 이 정도면 좋아요"
- 옵션: `5~10분 / 15~30분 / 1시간 이상 무관 / 즉흥적으로(제한없음)`
- `hobby_profile.세션당시간`과 매칭하는 **하드 상한 필터**로 적용.
- `burnout_severe` 선택 시 세션시간 미선택이어도 서버 기본값 `'15~30분'`을 강제 적용.

## 7. 하드필터 매핑 규칙 (예산·사회성·활동량·시간축 = 하드 제약)

```python
BUDGET_CEIL = {
  'budget_0':   ['무료'],
  'budget_low': ['무료','1만원 이하'],
  'budget_mid': ['무료','1만원 이하','3만원 이하'],
  'budget_high':['무료','1만원 이하','3만원 이하','3만원 이상'],
}
SOCIALITY = {
  'solo_strong': ['완전 개인'],
  'solo_mild':   ['완전 개인','온라인 느슨한 연결'],
  'flexible':    ['완전 개인','온라인 느슨한 연결','오프라인 소규모','오프라인 대규모'],
  'social_mild': ['온라인 느슨한 연결','오프라인 소규모'],
  'social_strong':['오프라인 소규모','오프라인 대규모'],
}
ACTIVITY = {
  'static':  ['정적'],
  'light':   ['정적','저강도'],
  'moderate':['저강도','중강도 이상'],
  'intense': ['중강도 이상'],
}
```

- `budget_0` 선택 시 지속비용=`정기결제`인 취미는 함께 제외.
- 크로노타입 `night_owl` + 가용시간대 `weekday_evening` 조합이면 공간=`외출 필요` 취미는 제외.
- **의외 추천도 이 하드 제약을 반드시 지킨다** (FR-407-1, 확정 사항. 취향/성향 차원에서만 의외성 부여, 실행 제약 무시는 미채택).

### 7-1. 모순 케이스 완화 (강도='정적' AND 사회성>=`social_mild`)
활동강도가 정적인데 사회성 선호가 높은 조합(FR-302 모순 플래그)은 그대로 필터하면 오프라인 소규모 취미(전부 강도 저강도 이상)가 전멸한다.

```
IF 모순_flag == True:
    강도 필터를 'static' → 'static + light' 로 1단계만 완화 (예산·시간·공간 등 다른 하드제약은 유지)
    카드 근거문장에 "저강도로 참여 가능한" 문구 명시
```

### 7-2. 후보 부족 시 완화 체인
```
IF 하드필터 통과 후보 수 < 3:
    1) 가장 최근 적용 제약(활동량 또는 사회성)부터 1단계씩 완화
    2) 그래도 3 미만이면 RIASEC 코드 매칭만으로 완화 검색
    3) 완화 카드에는 "선호와 100% 일치하진 않지만" 톤의 안내 문구 삽입
```
외부 LLM/임베딩 API 자체 장애 시에는 이 체인 대신 `fallback_정적세트`를 사용한다.

### 7-3. 카드 배분 우선순위 (안정 vs 의외 충돌 시)
```
1순위: 의외 카드 1개 확보 (협상 불가)
2순위: 남은 후보로 안정 카드, 최소 2개까지는 확보
3순위: 안정 카드 2개 미만이면 §7-2 완화 체인을 먼저 적용 후 재배분
```

## 8. RIASEC 추론 — 고정 매핑표 기반, LLM 자유 추론 금지

9축 설문은 RIASEC을 직접 묻지 않는다. "의외"의 기준이 세션마다 흔들리지 않도록 **규칙 기반 고정 매핑**을 사용한다 (코드/시트로 관리, 프롬프트에는 "아래 표 기준으로만 판단, 자유 추론 금지" 명시):

```
동기(motive_creation) → A +1
```

## 9. AI 파이프라인 (구현 순서 준수)

```
[Step1/2 응답]
→ ① 구조화 피처 변환 (설문 선택값 → user_profile 정형 필드, 단순 규칙 매핑)
→ ② LLM 해석 (JSON: motivation/emotional_state/constraints/risk_flag 산출)
→ ③ 후보 검색 (태그 매칭 1차 필터 → 임베딩 벡터 검색 ChromaDB)
→ ④ LLM 선별·설명 (3~5개 선택, 근거문장+첫행동+recommendation_type, 반드시 "의외" 1개 포함)
→ ⑤ RAG 보강 (비용/장비/난이도/시간/안전 — 지식베이스 조회값만, LLM 생성 금지)
→ [처방형 카드 3~5개]
   ├─ Step4 피드백 → ②~④ 1회만 재실행 (프로필 임시 보정, 그 이상 재요청 불가)
   └─ Step5 체크인 → user_profile 장기 필드 갱신
※ ② 단계에서 risk_flag=true면 ③~⑤ 즉시 미실행 → 공감 응답 + 상담 안내로 즉시 분기
```

### Step2 (적응형 대화) 발동 조건
- **설문상 모순 AND 시간부족/감정신호가 동시에 충족될 때만** 발동 (오탐·대화 피로 방지). 모순 없으면 생략하고 Step3 직행.
- 최대 **2~3턴** 상한 (내부 카운터, UI 미노출).
- 모든 질문은 "왜 이 질문을 하는지"가 설문 답변과 근거로 연결되어야 한다.

## 10. 카드 UI 규칙 (프론트 구현 시 준수)

- 카드 정보 표시 순서(확정): **근거문장 → 비용/시간 → 오늘의 첫행동 → 혼자·함께 + 안전근거 → 난이도 → 실행경로**.
- **일반(안정) 추천**과 **의외 추천**은 화면상 명확히 구분되는 별도 영역("나에게 맞는 추천" vs "뜻밖의 발견")으로 노출한다 (FR-508).
- 안전근거 문구는 **오프라인 소규모/대규모 추천에만** 필수 포함. 온라인/완전개인 추천엔 생략.
- 첫행동은 5~10분 내 완결 가능한 행동만, 예약·결제·가입 없음.
- 비용은 항상 "예상 범위" 고지 동반, 확정 가격 단정 표현 금지.
- 각 카드에 추천 타입(안정/의외/회복/관계) 라벨 표기, 의외 카드는 라벨 필수.

## 11. Fallback 전략 요약

| 상황 | 대응 |
|---|---|
| LLM JSON 스키마 미준수 | 재요청 1회 → 실패 시 설문 태그 매칭만으로 추천 생성 |
| 자유입력 비어있음/무의미 | 설문 데이터만으로 Step3 직행 (Step2 생략) |
| 벡터 검색 결과 0건 | RIASEC 코드 매칭 기준 완화 검색 |
| 외부 API(임베딩/LLM) 장애 | `fallback_정적세트`(RIASEC 유형별 대표 6개)로 즉시 대체 |
| 심한 소진·자해 신호 감지 | 추천 로직 즉시 중단, 공감+상담 안내 (최우선, 다른 fallback보다 상위) |

## 12. 성능 목표 (구현 시 참고, 부하테스트로 보정)

| 지표 | 목표 |
|---|---|
| Step1→Step3 추천 생성 (모순 없음 경로) | p95 ≤ 6초 |
| Step2 대화 1턴 응답 | p95 ≤ 3초 |
| Step4 재구성(1회) | p95 ≤ 5초 |
| 스트리밍 첫 토큰(TTFT) | ≤ 1.5초 (설명 생성은 스트리밍 권장) |
| 벡터 검색 단독 | ≤ 300ms |

비용/속도 원칙: 프롬프트에 불필요한 컨텍스트 금지(후보 취미 3~5개로 제한), Step4 재구성 1회 상한, risk_flag 분기 시 후속 LLM 호출 즉시 중단, fallback 정적 응답은 캐시로 서빙(LLM 호출 없음).

## 13. 보안 / 개인정보

- 모든 클라이언트-서버, 서버-외부API 통신 TLS 필수.
- 감정·번아웃 데이터(`energy_level`, `burnout_state`, `current_emotion`)는 민감정보로 분류 → **식별 정보와 별도 테이블 분리 저장**, 결합 조회 최소화, 최소 수집·명시적 동의 원칙.
- 자유입력 원문은 필요 목적 외 장기 보관 금지(보관 기간 정책 별도 확정 필요 — 임의로 장기 보관 로직 만들지 말 것).
- 진단형 필드명(`번아웃 점수`, `우울 진단`) 사용 절대 금지 → `energy_level`, `burnout_state` 등 참고형 명명만 사용.
- API 키는 환경변수/시크릿 매니저, 저장소에 커밋 금지.

## 14. Non-Goals (구현하지 말 것)

- 랭킹 스코어 산식, 전역 학습, 협업 필터링, 컨텍스추얼 밴딧, 자체 ML 학습
- 행동 기반 5~10분 미니체험 자동화 (Step5 체크인으로 대체됨)
- 결제·예약 직접 연동, 실제 리퍼럴 정산, 외부 플랫폼 API 연동
- 자동 푸시 알림 기반 체크인 (재방문은 완전 user-initiated)
- 수동 신호 연동(음악/영상/사진/캘린더/소비 카테고리 등)
- 커뮤니티 기능, B2B 인사이트 판매, 파트너 대시보드
- 취미 자체 판매·클래스 직접 운영
- 번아웃·감정 상태에 대한 의학적 진단
