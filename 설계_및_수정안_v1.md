기준 문서: **PRD 취미로 v1.3 · AI Prompt Spec v1.3 · hobby_profile_dataset_v2.json (78종)**

> **v2 갱신 요지 (PRD v1.3 반영):**

## 0. 두 프로젝트 진단 요약

| | **sesac_mini_project1** ("할래말래"/"취미로") | **hobbydna-mvp-poetry** (하비DNA) |
|---|---|---|
| 성격 | 기획·데이터는 우수, **구현은 20%** | 기획은 단순하지만 **작동하는 배포 직전 MVP** |
| 백엔드 | `schema.sql` + `seed_data.py` 뿐. **FastAPI 앱 없음** | FastAPI 완성: `main.py`+`auth.py`+`hobby_agents.py`(1063줄) |
| 인증 | 스키마만 있음(구글 로그인 미구현) | **구글 OAuth2 + JWT 실동작** (`auth.py`) |
| 추천엔진 | 코드 없음(파이프라인은 PRD 문서로만 정의) | 대화형 에이전트+랭커+안전가드+리포트 **작동** |
| 데이터 | **취미 78종(v2), 검증필드 풍부**(비용/안전/첫행동/RIASEC/RAG) + fallback 6종 | 취미 ~12종 코드에 하드코딩 |
| 프론트 | Stitch 생성 정적 HTML 13장, **백엔드 미연결** | 단일 SPA `index.html`(모바일 UI 완성), fetch 연동됨 |
| DB | Supabase(Postgres) + ChromaDB 설계 | SQLite(파일) + 인메모리 |
| 배포설정 | **없음** | Procfile·railway.json·runtime.txt·requirements.txt·vercel.json **모두 있음** |

**결론:** 둘은 사실상 같은 서비스의 두 버전입니다. sesac은 "좋은 재료(데이터·PRD)", hobbydna는 "작동하는 뼈대(배포·인증·API)". → **hobbydna의 검증된 뼈대를 sesac으로 이식하고, sesac의 40종 데이터·9축·의외추천 규칙을 얹는 것**이 2일 개발/비전공자 팔로업 제약에 가장 부합합니다.

## 1. 권장 아키텍처 (Option A — 이식·병합, 추천)

```
[브라우저]
   │  Google 로그인 버튼
   ▼
[Vercel] 정적 프론트 (index.html / config.js)   ← sesac frontend
   │  fetch(JWT Bearer)          ▲  #token= 리다이렉트
   ▼                             │
[Railway] FastAPI 백엔드 ────────┘             ← hobbydna backend 이식
   ├─ auth.py     : 구글 OAuth2 + JWT (그대로 재사용)
   ├─ main.py     : /session /message /recommend /feedback /library ...
   ├─ pipeline.py : 9축설문→(자유입력 원샷분석)→하드필터→완화체인→GPT선별(의외1개·1.5배완화)→RAG  ← 신규(PRD v1.3)
   ├─ data.py     : hobby_profile_dataset_v2.json 로드(78종+fallback 6종)  ← sesac 데이터
   ├─ OpenAI SDK  : 해석/선별/설명
   └─ Supabase(Postgres) : user_auth / user_profile / feedback_history / business_event
```

- **로그인은 구글 하나**: `auth.py`의 `/auth/google/login` → 구글 동의 → `/auth/google/callback` → JWT 발급 → 프론트 `#token=...`로 전달 → localStorage 저장 → 이후 모든 API에 `Authorization: Bearer`.
- **DB 선택**: Railway 디스크는 재배포 시 휘발 위험이 있어 **SQLite 대신 Supabase(관리형 Postgres) 권장**. sesac `schema.sql`이 이미 있으니 그대로 사용. (ChromaDB는 MVP에선 생략하고, 40종은 JSON 인메모리 + 임베딩 코사인으로 충분 — 규모가 작음.)

### Option B — PRD 원안 그대로 (Next.js + Supabase + ChromaDB)
PRD에 100% 충실하지만 Next.js 신규 구축 + ChromaDB 상시서버 운영으로 **작업량·인프라가 크게 늘어남**. 2일 MVP·비전공자 팔로업 원칙과 상충 → **비추천(2차 고도화 때 전환)**.

## 2. 반드시 해야 할 수정사항 (체크리스트)

### 2-1. 백엔드 (신규 생성 — sesac엔 앱이 없음)
1. `backend/main.py` 신규: hobbydna `main.py`를 기반으로, 엔드포인트를 sesac 플로우(설문 9축→추천→재구성→체크인)에 맞게 조정.
2. `backend/auth.py` 신규: hobbydna `auth.py` **거의 그대로 복사**. 구글 OAuth2+JWT.
3. `backend/pipeline.py` 신규(핵심): PRD v1.3 · Prompt Spec v1.3 추천 파이프라인 구현
   - 9축 설문(`user_survey_vector`) → `user_profile` 정형 필드 매핑 + 하드필터 규칙
   - **원샷 심층분석(FR-402)**: 자유입력(`user_input`, 선택)이 있으면 GPT-5.4-mini가 **한 번에** 이원 분석
     — (a) `hobby_signal_analysis`(RIASEC 신호·선호활동, 내부 가중치로만 사용)
     — (b) `user_status_analysis` = `{motivation, emotional_state, constraints, risk_flag}`
       · `emotional_state` ∈ `steady / burnout_type / boreout_type / none_detected`
       · `none_detected`이면 9축 벡터 기반 매핑으로 대체
   - **모드 3종 출력**: `result`(정상) / `clarify`(보완질문 **최대 1회**, `clarification_state.asked`로 추적 — 답변 후엔 무조건 result/risk로 종료, **다중턴 금지**) / `risk`
   - `risk_flag`(자해/극단 무기력) 감지 → 모든 로직보다 최우선, 즉시 중단 + 상담 안내 문구(고정). **risk는 자유입력/보완질문 답변에서만 감지**(벡터만으로 판별 안 함)
   - 하드필터(예산·시간·공간·사회성·활동량) → 후보 부족 시 **완화체인**(최근 제약 완화 → RIASEC 매칭 → `fallback_정적세트` 6종)
   - GPT 선별로 3~5개, **의외 추천 1개 항상 포함**. 의외 추천만 **하드 실행제약 절반 완화(FR-407-1): 제약치의 최대 1.5배까지 허용**(예: 예산 1만원→1.5만원). 일반 추천은 하드제약 100% 유지. 취향(RIASEC·감각·신규성) 의외성은 그대로.
   - 설명은 RAG설명 필드 사용, **비용·안전은 LLM 생성 금지·데이터셋 값 사용**
   - ※ v1.2의 `conversation_turns`(다중턴) 구조는 **폐기** — Prompt Spec v1.3에서 삭제됨.
4. `backend/data.py` 신규: `hobby_profile_dataset_v2.json`(**78종** + fallback 6종) 로드 + `is_synthetic=True` 분리 + 임베딩(옵션).
5. 금지 잔재 제거: `pymupdf`/`pdfplumber`/`python-multipart` 미사용, "계약서" 템플릿 흔적 배제(PRD에 명시됨).

## 3. Supabase vs SQLite 결정 근거
- Railway 컨테이너 파일시스템은 재배포/재시작 시 초기화될 수 있어 SQLite 파일(`hobbydna.db`) 데이터가 사라질 위험.
- 실제 사용자 로그인/피드백을 남기려면 **관리형 Postgres(Supabase)**가 안전. sesac이 이미 5개 테이블 스키마를 설계해 둠 → 그대로 채택.
- 단, 초스피드 데모만 필요하면 SQLite로 시작 후 Supabase로 승격 가능(코드상 저장계층만 교체).

## 4. 단계별 실행 순서 (제안)
1. **정리**: sesac에 `backend/` 앱 골격 + 배포파일 6종 추가, 깨진 git 폴더 삭제.
2. **인증**: `auth.py` 이식 → 로컬에서 구글 로그인 왕복 확인(`DEV_NO_AUTH`로 우회 테스트 병행).
3. **데이터/엔진**: `data.py`(40종) + `pipeline.py`(9축·하드필터·완화·의외1개·안전) 구현, 오프라인 폴백 유지.
4. **API**: `/session /survey /recommend /feedback /report` 정리. `/recommend`는 `{user_survey_vector, user_input?, clarification_state?}`를 받아 `mode: result|clarify|risk` 로 응답(원샷, 다중턴 없음).
5. **프론트 연결**: 로그인 버튼 + 설문 제출 + 추천 카드(안정/의외 분리) + 피드백.
6. **배포**: Railway(백엔드)→URL 확보→`config.js`/`REDIRECT_URI`/`FRONTEND_URL` 채우기→Vercel(프론트)→구글 콘솔 리디렉션 등록→E2E 로그인·추천 검증.
7. **검증**: 추천 생성 p95 ≤ 6초, LLM JSON 실패 시 1회 재시도+fallback, **의외추천 항상 1개 노출**, **의외추천 1.5배 완화 제약 준수(일반은 100%)**, **보완질문은 최대 1회 후 종료(다중턴 없음)**, risk 감지 시 즉시 안내 전환 테스트.

## 5. 핵심 리스크 & 방어
- **의외추천 누락**: 파이프라인 마지막에 "의외 카드 1개 없으면 강제 삽입" 단정 검사. 화면은 안정 vs "뜻밖의 발견" 영역 분리(FR-508).
- **의외추천 제약 위반**: 의외 후보는 하드제약 1.5배 이내인지 검사(FR-407-1). 일반 추천은 1.0배 초과 시 제외.
- **다중턴 재발 방지**: `clarification_state.asked==true`면 어떤 경우에도 `clarify` 재반환 금지 → 무조건 `result`/`risk` 종료.
- **비용/안전 할루시네이션**: 해당 필드는 LLM 출력에서 배제하고 데이터셋 값만 렌더.
- **키 노출**: LLM 호출은 반드시 백엔드 경유, 프론트에 어떤 API 키도 두지 않음.
- **모델명**: PRD의 `gpt-5.4-mini`/임베딩 모델명은 착수 시점 최신 가용 모델로 재확인 후 env로 주입.
