# 취미로 — 팀 작업 가이드 (하비DNA 에이전트 이식 완료판)

이 문서 하나로 **백엔드 담당 / 프론트 담당**이 각자 자기 Claude Cowork에서 바로 작업할 수 있습니다.
하비DNA(HobbyDNA)의 작동하는 에이전트가 **이미 이 저장소 안에 이식**되어 있습니다. (아래 "이식된 파일" 참고)

---

## 0. 지금 상태 한눈에

- 하비DNA의 **작동하는 백엔드/프론트가 sesac 저장소에 복사 완료** → 깃허브에 push하면 팀 전원이 받습니다(별도 hobbydna 파일 불필요).
- 이 상태로 **바로 로컬 실행·배포 가능**(취미 12종 내장). 이후 우리 데이터 78종·PRD v1.3 규칙으로 진화시킵니다.
- 로그인은 **구글 하나**, 프론트=Vercel, 백엔드=Railway.

### 이식된 파일 (새로 들어온 것)
```
backend/
  main.py            # FastAPI 엔드포인트 (세션/메시지/추천/피드백/홈/보관함)
  auth.py            # 구글 OAuth2 + JWT 로그인  ← 그대로 사용
  hobby_agents.py    # 에이전트 본체(대화·추천·자기개선·안전·리포트)
  requirements.txt   # pip 의존성
  Procfile / railway.json / runtime.txt   # Railway 배포 설정
  .env.example       # 환경변수 템플릿 (복사해서 .env 만들기)
  schema.sql / seed_data.py   # (기존) Supabase 승격용
frontend/
  index.html         # 바로 동작하는 단일 SPA (채팅·리포트·피드백 UI)
  config.js          # 백엔드 주소 설정
  vercel.json        # Vercel 배포 설정
  src/               # (기존) Stitch 디자인 13화면 — 참고/이식용
data/
  hobby_profile_dataset_v2.json   # 취미 78종 (우리 데이터, 진화 목표)
  PRD_취미로_v1.3.md / 취미로_AI_Prompt_Spec_v1.3.md
```

## 1. 모두 공통 — 최초 1회 세팅 (git + Cowork)

1. **저장소 받기**: 깃허브 sesac 저장소를 clone (또는 이미 있으면 `git pull`).
2. **Cowork에서 이 폴더를 연결**: Claude Cowork에서 clone한 폴더를 작업 폴더로 선택.
3. **브랜치 나눠 쓰기**(충돌 방지): 각자 자기 브랜치에서 작업 후 PR.
   - 백엔드: `feat/backend-...`
   - 프론트: `feat/frontend-...`
4. **`.env`는 절대 커밋 금지** — 이미 `.gitignore`에 있음. 각자 `backend/.env.example`을 복사해 `.env`로 만들고 값 채움.
5. **깨진 폴더 삭제**(있으면): `.git_broken_delete_me`, `.git_broken_delete_me2` 폴더는 로컬 탐색기에서 수동 삭제.

> Cowork 협업 팁: 각자 자기 브랜치에서 Claude에게 "이 파일 수정해줘"라고 지시 → Claude가 파일 편집 → 본인이 `git add/commit/push` → 깃허브에서 PR 리뷰·머지. 서로 다른 파일을 만지면 충돌이 거의 없습니다. **역할별로 담당 폴더가 갈려 있어(백엔드=backend/, 프론트=frontend/) 병행 작업이 안전합니다.**

## 2. 로컬에서 돌려보기 (5분)

**백엔드**
```bash
cd backend
python -m venv .venv && source .venv/bin/activate   # (Windows: .venv\Scripts\activate)
pip install -r requirements.txt
cp .env.example .env      # DEV_NO_AUTH=1 이면 로그인 없이 테스트 가능
uvicorn main:app --reload --port 8000
# 확인: http://localhost:8000  →  {"ok": true} 나오면 성공
```

**프론트**
```bash
cd frontend
# config.js 의 window.HOBBY_API 가 http://localhost:8000 인지 확인
python -m http.server 3000     # 아무 정적 서버나 OK
# 브라우저: http://localhost:3000
```

`DEV_NO_AUTH=1`이면 구글 로그인 없이도 추천 흐름을 바로 볼 수 있습니다. 실제 로그인 붙일 땐 4장(배포) 참고.

## 3. 백엔드 담당이 할 일 (backend/ 폴더)

순서대로. 각 항목은 Claude Cowork에서 "○○ 해줘"로 맡기면 됩니다.

1. **[필수] 우리 데이터 78종 연결** — `hobby_agents.py`의 `HOBBY_CATALOG`가 지금 12종 하드코딩입니다.
   → `data/hobby_profile_dataset_v2.json`을 읽어 카탈로그를 만드는 로더(`backend/data.py`)를 작성하고 교체.
   → 필드 매핑: `취미명, riasec_code, 초기비용/지속비용, 세션당시간, 공간, 사회성, 강도, beginner_action, safety_notes, RAG설명` 등.
2. **[필수] PRD v1.3 추천 규칙 반영** — Prompt Spec v1.3 기준:
   - **원샷 심층분석(FR-402)**: 자유입력 1개 → GPT가 (a)취미신호 +(b)유저상태 이원 분석. **다중턴 대화 없음**.
   - **모드 3종**: `result / clarify(보완질문 최대 1회) / risk`. `clarify`는 1회 후 무조건 종료.
   - **의외 추천 1개 항상 포함** + 의외만 하드제약 **1.5배까지 완화(FR-407-1)**, 일반은 100% 유지.
   - **risk 최우선**: 자해/극단 무기력 신호 시 추천 중단 + 상담 안내 문구.
   - 비용·안전 필드는 **LLM 생성 금지**, 데이터셋 값만 사용.
3. **[필수] 구글 로그인 확인** — `auth.py`는 그대로 동작. 로컬에서 `DEV_NO_AUTH=0`으로 두고 구글 왕복 테스트(4장 준비 필요).
4. **[선택/승격] SQLite → Supabase** — 지금은 `hobbydna.db`(SQLite). 실서비스 데이터 영속화가 필요하면 `schema.sql`로 Supabase 만들고 저장계층만 교체.
5. **[배포] Railway** — `Procfile/railway.json/runtime.txt` 이미 있음. 4장 참고.

**백엔드가 만지는 파일**: `backend/*.py`, `backend/requirements.txt`, `backend/*.json/Procfile/runtime.txt`. 프론트 파일은 건드리지 않기.

## 4. 프론트 담당이 할 일 (frontend/ 폴더)

1. **[필수] 백엔드 주소 설정** — `frontend/config.js`의 `window.HOBBY_API`를 로컬은 `http://localhost:8000`, 배포는 본인 Railway URL로.
2. **[필수] 로그인 버튼 연결** — 첫 화면 버튼 → `location.href = window.HOBBY_API + "/auth/google/login"`.
   콜백 후 주소창 `#token=...` 파싱 → `localStorage`에 저장 → 모든 fetch에 `Authorization: Bearer <token>` 헤더. (index.html에 이미 패턴 있음, 참고)
3. **[필수] 화면 = PRD v1.3 규칙 반영**:
   - 자유입력은 **선택**. 안내 문구: **"더 상세한 분석을 원하신다면 입력해주세요"**.
   - 결과 화면은 **안정 추천 + "뜻밖의 발견"(의외) 영역 분리** 노출.
   - 보완질문(`clarify`)은 **1회성** 화면(재질문 없음).
   - 금지단어 카피 점검: "진단/점수/장애/우울증" 사용 금지.
4. **[선택] 디자인 이식** — 지금 베이스라인은 하비DNA `index.html`. 우리 Stitch 디자인(`frontend/src/`의 13화면)을 입히고 싶으면, 각 화면에 위 fetch 연동을 붙여 교체.
5. **[배포] Vercel** — `vercel.json` 이미 있음. 4장 참고.

**프론트가 만지는 파일**: `frontend/*`. 백엔드 파일은 건드리지 않기.

## 5. 배포 (프론트=Vercel, 백엔드=Railway, 로그인=구글)

**A. 구글 클라우드 콘솔** (로그인 담당 1명이 대표로)
1. OAuth 동의화면 구성(외부, 테스트 사용자 등록).
2. OAuth 클라이언트 ID(웹) 발급 → `GOOGLE_CLIENT_ID/SECRET` 확보.
3. 승인된 리디렉션 URI에 `https://<railway-url>/auth/google/callback` 등록(로컬용 `http://localhost:8000/...`도 함께).

**B. Railway (백엔드)**
1. Railway에서 GitHub 저장소 연결, **루트 디렉터리를 `backend/`로 지정**.
2. 환경변수 등록: `OPENAI_API_KEY, HOBBY_GPT_MODEL, GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, REDIRECT_URI(=Railway URL 기준), FRONTEND_URL(=Vercel URL), JWT_SECRET, DEV_NO_AUTH=0`.
3. 배포 후 나온 URL 확보(예: `https://xxx.up.railway.app`).

**C. Vercel (프론트)**
1. Vercel에서 같은 저장소 연결, **루트를 `frontend/`로 지정**.
2. `config.js`의 `HOBBY_API`를 Railway URL로 바꿔 커밋 → 배포.
3. 나온 Vercel URL을 Railway의 `FRONTEND_URL`과 구글 콘솔에 반영(서로 주소를 알아야 CORS·리다이렉트가 맞음).

**D. 최종 점검**: Vercel 사이트 접속 → 구글 로그인 → 설문 → 추천 카드(안정+의외) → 피드백까지 왕복 확인.

## 6. 절대 규칙 (전원)
- `.env`·API 키는 **깃허브에 올리지 않는다**(대시보드 환경변수로만).
- LLM 호출은 **반드시 백엔드 경유**. 프론트에 키를 두지 않는다.
- 비용·안전 정보는 **데이터셋 값**만 표시(LLM이 지어내지 않게).
- 의외 추천은 **항상 1개**, 화면에서 일반 추천과 분리.
- 자기 담당 폴더 위주로 작업(백엔드=backend/, 프론트=frontend/) → 충돌 최소화.
